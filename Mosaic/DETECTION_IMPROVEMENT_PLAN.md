# 自動認識改善検討書（複数人物の乳首・性器検出）

- 作成日: 2026-07-22
- 作成者: Claude Code（技術サーベイはAIの学習知識 2025年2月時点まで。以降の新規発表は別途Web調査で補完すること）
- 位置づけ: `Mosaic/ARCHITECTURE.md` §5.2 の課題（固定比率ヒューリスティック・単一人物・カテゴリ非対応）を解消するための改善設計。実装前の検討資料であり、本書自体は仕様変更ではない。

## 1. 現状の課題整理

| 項目 | 現状 | 問題 |
|---|---|---|
| 人物検出 | `VNGeneratePersonSegmentationRequest` の結果を真偽判定のみに使用、矩形は固定比率 | 実位置を検出していない。複数人物は常に1件扱い |
| 骨格推定 | 人物矩形からの固定比率算術 | 関節情報なし。姿勢変化に全く追従しない |
| ROI生成 | 下半身固定比率、confidence 0.42固定 | 乳首（胸部）を対象にできない。カテゴリ非対応 |
| 詳細候補 | パススルー | 部位単位の検出が存在しない |
| マスク | 図形 or Vision人物マスク切出し（Build 8で接続済み） | 部位単位の画素マスクは未対応 |
| ドメイン | — | 検証画像は漫画。実写学習済みモデルはそのままでは漫画に弱い（ドメインギャップ） |

## 2. 技術サーベイ要約（2025年2月時点の知見）

### 2.1 Apple標準（外部モデル不要・最優先で活用すべき層）

- **`VNGeneratePersonInstanceMaskRequest`**（macOS 14+）: 人物を**インスタンス単位**（最大4人）で分離した画素マスクを返す。現在使用中の `VNGeneratePersonSegmentationRequest`（全人物合成1枚）と異なり、複数人物のグループ分けに直結する。
- **`VNDetectHumanBodyPoseRequest`**（macOS 11+）: **複数人物対応**の2D骨格（19関節: 肩・肘・腰・膝等）。保留中の「骨格ボーン表示」の実データ源になり、肩→胸部、腰→鼠径部という解剖学的プライア（事前分布）でカテゴリ別ROIを合理的に生成できる。
- **`VNDetectHumanRectanglesRequest`**: 人物矩形検出。インスタンスマスクの4人上限を超える場合のフォールバック。
- **SensitiveContentAnalysis framework**（macOS 14+）: オンデバイスのヌード判定。ただし**画像単位の判定のみで位置は返さない**ため、処理要否のトリアージ（無関係画像のスキップ）用途に限る。

### 2.2 部位検出（NSFW object detection）

- **NudeNet**（notAI-tech）: 乳首・性器（男女別）・臀部等をバウンディングボックスで返すオープンソース検出器の代表格。v3はONNXで配布されておりONNX Runtime/CoreML変換の実績が多い。**ライセンスはバージョンごとに要確認**（配布形態に影響）。
- **Bumble private-detector**: 分類のみ（位置なし）。トリアージ用途。
- CLIP系NSFW分類器（LAION safety、SD safety checker等）: いずれも分類のみで位置情報なし。
- 結論: 位置つき部位検出の既製モデルは実質NudeNet系のみ。品質・ライセンス・漫画対応を考えると**自前ファインチューニングを本命**とすべき。

### 2.3 汎用検出器（自前学習のベース）

- **YOLOv8/v9/v10/YOLO11**（Ultralytics系）: 学習容易・軽量だが**AGPL-3.0**。アプリ同梱配布ではライセンス波及リスクがあるため注意。
- **RT-DETR / RTMDet**（Apache-2.0）: ライセンスが安全な高精度検出器。RTMDetは軽量版（tiny/s）でリアルタイム可。**ライセンス面から本命候補**。
- **Grounding DINO / YOLO-World**（CVPR 2024）: オープン語彙検出。テキストプロンプトで未学習カテゴリを検出できるが、NSFW部位の語彙は安全フィルタで弱く、実用にはやはりファインチューニングが要る。
- 小物体対策: 乳首は画像全体比で極小。**SAHI**（スライス推論）や、人物クロップ→高解像度再推論の2段構成が定石。全身画像への一発推論は再現率が大きく落ちる。

### 2.4 画素マスク（モザイク形状の精密化）

- **SAM / SAM 2**（Meta, 2023/2024）: ボックスプロンプトから高品質マスク。**MobileSAM / EfficientSAM / FastSAM** はオンデバイス向け軽量版で、CoreML変換実績もある。部位ボックス→SAMマスク→モザイクの「Grounded-SAM」型パイプラインは2023-24年の事実上の標準構成。
- 現行 `Segmenting` プロトコル（Build 8で `[CIImage]` マスクを返す形に刷新済み）は、SAM系マスクをそのまま差し込める境界になっている。

### 2.5 骨格推定（Vision以外の選択肢）

- **RTMPose / RTMO**（OpenMMLab, Apache-2.0）: 複数人物・高精度・軽量。Visionの精度が不足する場合の置換候補。
- **MediaPipe Pose**: 単人向け設計のため複数人物用途では人物検出との組合せが必要。優先度は低い。

### 2.6 漫画・イラストのドメインギャップ

- 実写学習済み検出器は線画・スクリーントーンの漫画でほぼ機能しない。これは学術的にも既知の問題（実写→イラストのドメイン適応）。
- 対策の実務的序列: (1) **漫画・イラストの自前アノテーションデータでファインチューニング**（最も確実）、(2) 実写+イラスト混合学習、(3) スタイル変換によるデータ拡張。
- アニメ系タガー（DeepDanbooru, WD14等）は分類のみで位置なし。データセットの自動事前ラベリング（画像選別）には使える。
- **データ整備上の注意**: 学習データは成人向けかつ権利・法令上問題のない素材に限定し、取り扱い方針を文書化すること。実写データは特に法的リスク審査を必須とする。

## 3. 提案アーキテクチャ（既存プロトコル境界へのマッピング）

現行の交換可能境界をそのまま使い、実装だけを段階的に差し替える。

| 境界 | 現行 | 提案 |
|---|---|---|
| `PersonDetecting` | 固定比率 | `VNGeneratePersonInstanceMaskRequest`（≤4人、超過時は人物矩形検出でフォールバック）→ 人物ごとの実マスク+実矩形 |
| `PoseEstimating` | 固定比率 | `VNDetectHumanBodyPoseRequest`（複数人物）。関節座標を保持する形へ `PoseHint` を拡張（骨格ボーン表示の保留解除に直結） |
| `ROIGenerating` | 下半身固定比率 | 関節からの解剖学的プライア: 胸部=肩と腰の中間帯（乳首カテゴリ）、鼠径部=左右腰関節の下方（性器カテゴリ）。人物ごと・カテゴリごとにROIを生成し `MosaicROI.category` へ設定 |
| `CandidateDetecting` | パススルー | 部位検出器（Phase 2）: 人物クロップを高解像度で推論し、乳首/性器（男女）ボックス+信頼度を返す。プライアROIとの和集合を取り**再現率優先**で統合 |
| `Segmenting` | 図形/Vision人物 | Phase 3でSAM系（ボックスプロンプト→部位画素マスク）を第3の選択肢として追加。UIの「マスク生成」切替に統合 |
| 人物への帰属 | — | 部位ボックス中心が属する人物インスタンスマスクで帰属判定 → 既存の「人物N」レイヤ自動グループへ接続 |

## 4. 段階的ロードマップ

### Phase 1: Apple標準APIの実活用（外部モデル不要・最優先）
- インスタンスマスク+複数人物骨格で「常に1件・固定比率」を解消。UIの複数人物グループ化・骨格ボーン表示・人物シルエット破線表示（保留2件）の実データが揃う。
- 工数目安: 小〜中。プライバシー: 完全ローカル（Vision）。**費用対効果が最も高く、即着手を推奨**。

### Phase 2: 部位検出モデルの導入（カテゴリ別自動認識の本丸）
- RTMDet/RT-DETR系（Apache-2.0）を漫画+実写アノテーションデータでファインチューニングし、CoreML変換してオンデバイス実行。人物クロップ2段推論+SAHIで小物体対応。
- `MosaicTargetCategory` へ自動でカテゴリ設定。ユーザー提供予定の参照画像はこのフェーズの学習・評価データとして使用。
- 工数目安: 大（データ整備が主工数）。モデル同梱/初回ダウンロードの方式は実装前にユーザー確認（CLAUDE.md §2.2）。

### Phase 3: 部位画素マスク（モザイク形状の精密化)
- MobileSAM/EfficientSAM系をCoreML化し、部位ボックスをプロンプトに画素マスク生成。「マスク生成」ポップアップの第3項目として追加。

### Phase 4: ローカル継続改善
- `HistoryEngine`/ライブラリに蓄積されるユーザー修正（追加・削除・移動・リサイズ）を弱教師データとしてローカルで活用（能動学習）。外部送信はしない（ARCHITECTURE §5遵守）。

## 5. 評価基準（案）

- ローカル評価セット（漫画/実写×単人/複数人）を作成し、カテゴリ別**再現率を最優先**（見逃し=モザイク漏れが最重リスク。過検出は半自動UIで削除可能）。
- 目標例: IoU 0.3で乳首・性器の再現率 ≥ 0.9（モザイク用途では緩いIoUで十分）。誤検出は1画像あたり許容数を別途設定。
- 処理時間: ARCHITECTURE §6の「3秒以内」を維持（Visionインスタンスマスク+骨格で数百ms、クロップ部位検出で数百ms、合計1秒前後を想定）。

## 6. 実装ステータス

- Phase 1: **実装済み**（2026-07-22, Build 10）。`VNGeneratePersonInstanceMaskRequest` + `VNDetectHumanBodyPoseRequest` + 解剖学的プライアROI生成。骨格ボーン表示・人物シルエット表示の保留も解除。
- Phase 2: 未着手
- Phase 3: 未着手
- Phase 4: 未着手

## 7. リスク・留意事項

- `VNGeneratePersonInstanceMaskRequest` は最大4人。5人以上は矩形検出フォールバックで縮退。
- Ultralytics系（YOLOv8以降）はAGPL-3.0。配布形態によってはソース開示義務が波及するため、Apache-2.0系（RTMDet/RT-DETR）を優先。NudeNetを使う場合もバージョンごとのライセンス確認を必須とする。
- 学習データの取り扱い（法令・権利・保管）は実装前に方針文書化が必要。
- 全推論はオンデバイスを堅持（ARCHITECTURE §5）。モデルの外部ダウンロードを追加する場合も画像データは一切送信しない。
